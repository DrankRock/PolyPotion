A NEW, ADDITIVE AUDIT · NOT A DECOR MAKER ROADMAP

The browser is already a small DCC. Treat it like one.

PolyPotion has crossed the interesting threshold: it is not a collection of toy viewers. It already has a local-first character pipeline, real mesh surgery, WebAssembly kernels, a rig/skin worker, retargeting, face work, a non-linear timeline, a stage, and two routes to video. The opportunity is now to make its pieces compose into a character performance and lightweight filmmaking machine—while exploiting browser primitives that desktop tools do not get for free.

Scope decision: this deliberately does not recommend a decor/environment maker. Props may enter a shot as imported GLB assets, but the ideas below focus on characters, rigs, meshes, animation, simulation, capture, presentation, and local collaboration.

WHAT WAS ACTUALLY REVIEWED

Repository read: strong substrate, a few deliberate seams

This audit was written against the root tools and documentation, all engine/worker definitions under studio3D_scripts/, AutoRig scripts, the PWA/service-worker policy, security headers, and the existing consolidated audit and progress tracker. It is a forward-looking companion, not a re-listing of already completed work.

EXISTS

Character pipeline is unusually complete

  • AutoRig + worker-based skin binding, weight paint, pose IK/poles/pins, bone conventions, retargeting and VRM export.
  • Morph/ARKit 52, landmark fitting, local face capture, lipsync and expression storage.
  • Motion, curves, timeline, VAT, and export formats already form a credible animation path.
EXISTS

Mesh craft is real, not cosmetic

  • Doctor, edit, sculpt, remesh, decimate/LODs, retopo, UV, bake maps, Boolean and texture work.
  • QEM, voxel remesh, BVH/raycast baking, Manifold CSG, shared symmetry and colour contracts.
  • Recipe gives the project a seed of reproducible, headless GLB processing.
EXISTS

Filmmaking has already begun

  • Stage assembles several animated actors and exports one glTF.
  • Director cuts camera sequences and records local WebM.
  • Physics also has real-time capture plus offline MP4/GIF paths using WebCodecs where available.
WATCH

The system is still page-per-tool

GLB transferables are a good pragmatic bridge, but each iframe owns a renderer and transient editing state. That makes a shared shot, shared animation data, and long jobs harder than they need to be.

THE PRODUCT SHAPE

Stop adding isolated editors. Build three compound workflows.

NOW

1. “Make this character perform.”

Import or generate → repair/retopo → rig → correct weights → create expressions → capture/author motion → clean it → retarget/export. This is PolyPotion’s most defensible path because most ingredients already exist.

DO → Turn handoffs into named, reversible “takes”: a pose, motion clip, face clip, audio clip, and export preset should survive as attached project metadata rather than vanish when one iframe closes.
NOW

2. “Make a 10–60 second character shot.”

Stage is the blocking room and Director is the camera room. Add a shot asset model—actors, takes, cameras, audio, cuts and render settings—and the suite becomes a local machinima/previs tool, not a decor tool.

DO → Keep it deliberately small: one scene, a few actors, imported props, a single edit timeline. Do not become Blender’s scene editor.
SPARK

3. “Let the web be the accelerator and the stage.”

WASM/Workers make formerly painful geometry interactive; WebGPU opens GPU compute; WebCodecs turns canvases into actual media; WebXR puts the character in a room; WebRTC makes review possible without a product backend.

DO → Design each capability as progressive enhancement: excellent local WebGL/JS baseline, faster WASM workers, then WebGPU/WebCodecs/XR where the browser grants it.
RIGGING & CHARACTER SYSTEMS

The next rigging leap is not another skeleton: it is rig behaviour.

AutoRig, skinning, weight paint, retarget and viewport IK are already the hard foundation. The biggest remaining value is making rigs produce believable motion quickly, survive non-humanoids, and export predictably.

NOW

Corrective shapes driven by joints

Elbows, shoulders, hips and knees collapse because weights alone cannot preserve volume. Let users paint/sculpt a corrective morph while a joint is posed, then bind the morph to an angle/range with a small driver graph. Export either baked morph animation or compatible glTF expressions.

DO → Start with pose-space correctives for elbow/knee/shoulder. Store {bone, axis, min, max, morph}; preview in Pose; bake keys on export. It joins existing Sculpt/Morph/Pose instead of inventing a rig language.
PRO

Control rig layer: FK/IK switches, spaces, constraints

Today’s IK handles are useful direct manipulation. A control layer makes animation authorable: FK↔IK matching, hand/foot/world space switching, parent/aim constraints, copy/paste/mirror controls, and a clean animator-facing naming scheme.

DO → Represent controls as metadata over the existing deform skeleton. Keep the export skeleton clean; bake control results to bone transforms. A minimal constraint evaluator is enough before attempting a Blender-style node rig.
SPARK

Browser-native “calibrate yourself” mocap

Use webcam pose capture plus a calibration pose, floor estimate, body-scale solve and contact detection to generate a reusable performer profile. Then apply temporal filtering, feet lock, root-motion extraction and confidence colouring before retargeting.

DO → Make the output honest: show tracked confidence, missing joints, contact markers and a before/after graph. A good cleanup pass matters more than chasing perfect markerless capture.
PRO

Creature / prop rig templates

Humanoids are covered. Expand sideways, not into generic arbitrary rigging: quadruped, winged creature, tail/spine chain, mechanical hinge/slider, and simple vehicle wheel templates. Each can map to the same solver/constraint primitives.

DO → Add template-specific landmark guides and export labels. A tail dynamics chain plus secondary spring bake would instantly broaden the suite without needing an all-purpose armature editor.
MESH, SCANS & SURFACE INTELLIGENCE

Make geometry diagnosis automatic, then make fixes explainable.

NOW

Character readiness report

Mesh Doctor already detects geometric failures and Export validates scale. Unite these into a “Can this become an avatar/game character?” report: watertightness, UVs, texture resolution, poly budget, bone count, weight normalization, blendshape naming, bounds, draw calls, mobile/XR estimates and target-specific warnings.

DO → Offer profiles—VRM/VTuber, mobile game, PC game, 3D print, film preview—and link each red line directly to the tool that resolves it.
PRO

Scan-to-character assistant

The project has repair, voxel remesh, retopo, UV, bake, rig and scale validation. Package them as a guided scan pipeline: orient → remove floating islands → fill/close → remesh → guide retopo → project/bake details → UV → rig readiness.

DO → Build it as a Recipe template first, with checkpoints and previews. Do not promise one-click perfect retopology; make the human decision points explicit.
SPARK

Semantic mesh selections

“Select left arm”, “all loose islands”, “inside mouth”, “thin parts”, or “high curvature” should become reusable sets. They power weight masking, materials, corrective-shape regions, physics pins, LOD protection and batch recipes.

DO → Start deterministically with topology/geometry predicates and named selection sets. Later, an opt-in local segmentation model can propose regions, but never make the pipeline depend on AI.
PRO

Texture-set discipline

Texture Paint, Bake Maps, atlas and colour tagging are a strong base. The missing production unit is a named texture set: base color, normal, ORM, emissive, opacity, thickness, masks; each with resolution, colour space, packing and UDIM-like tile awareness.

DO → Keep glTF as the canonical export target, but introduce a texture-set manifest in the project. It makes baking, shader work, compression and target validation coherent.
ANIMATION, STAGE & SMALL MOVIES

The strongest “insane web” feature is a pocket character film studio.

Director already records WebM, Physics can offline-render MP4/GIF with WebCodecs, Timeline assembles motion, Lipsync produces mouth tracks, and Stage owns multiple actors. The missing element is not rendering technology—it is a small shared editorial model.

NOW

Shot timeline: picture, performance and sound

Create a project-level sequence that references Stage actors and their clips, Director camera shots, Lipsync audio/takes, captions and render settings. A sequence should scrub deterministically and re-open exactly as it was.

DO → Use JSON metadata plus referenced GLB buffers; do not try to embed a full nonlinear editor. v1 needs cut/trim/fade, audio waveform, clip offset, camera cut, and a final render queue.
PRO

Offline frame rendering with WebCodecs

Real-time MediaRecorder is convenient but cannot guarantee frames. The Physics engine proves the project can render frame-by-frame and encode with VideoEncoder. Generalize that into a renderer that advances animation by exact frame time, renders a target canvas, feeds VideoFrames to an encoder, and muxes a file.

DO → Ship deterministic WebM first, add MP4 where AVC support/muxing is available, and retain real-time WebM as a fallback. Surface codec/browser support before a render begins.
SPARK

Performance takes: face + body + voice in one pass

Record a local session with webcam body landmarks, face blendshape coefficients and microphone audio under one clock. Permit independent re-timing, smoothing and replacement before baking all three streams to the character.

DO → Store compact landmark/coefficients/PCM-derived metadata—not raw camera video by default. It is smaller, privacy-aligned, editable, and preserves the “nothing uploaded” promise.
PRO

Animation cleanup toolbox

Motion capture becomes usable through filters: Butterworth-like smoothing, key reduction, root trajectory edit, foot contact locks, loop matching, motion trails, ghosting, joint-limit warnings and curve presets. These are boring compared with capture, but turn capture into work.

DO → Put the algorithms in headless Recipe-compatible functions so they can operate on a clip in Timeline, MoCap, Retarget or a batch recipe.
THE WASM MACHINE ROOM

WASM should become PolyPotion’s compute contract, not a collection of one-off tricks.

The hand-authored sparse gather kernel powers subdivision/relaxation and the XPBD solver handles cloth/soft-body constraints. The repository also already has a skin worker, UV worker, COOP/COEP credentialless deployment headers, and a visible threads state. This is enough to define a real compute strategy.

NOW

Define a job protocol and worker pool

Every expensive operation should accept {input, params, signal, onProgress} and return typed outputs. Put parsing, recipe runs, decimation, baking, remesh, exports and video frames behind cancellable jobs. UI tools only observe status and own previews.

DO → Create compute-worker.js / a small pool before importing a giant library. Use transferable ArrayBuffers by default; use SharedArrayBuffer only for cases with demonstrated benefit and a graceful 1-thread fallback.
PRO

Use compiled geometry libraries selectively

Potential candidates are meshoptimizer (reorder/simplify), xatlas (UV unwrap), Manifold (already conceptually present for CSG), Recast navigation, OpenVDB-like sparse fields, and robust mesh repair/parameterization libraries. WASM gives speed and battle-tested numerics, but not automatic product quality.

DO → Vendor pinned binaries/source locally, document license and feature detection, isolate their data conversion behind adapters, and benchmark each against the existing JS implementation before accepting bundle weight.
SPARK

GPU compute experiments that fit the suite

WebGPU compute can run a signed-distance volume, particle neighbour grid, texture dilation, skinning preview, morph accumulation, cloth constraints or massive crowd/VAT previews. It is ideal for “preview fast, bake exact” workflows.

DO → Do not rewrite renderers for WebGPU. Build one isolated compute-backed tool first—GPU SDF remesh or high-count particle cloth—with WebGL/WASM fallback.
WEB-ONLY SUPERPOWERS

The web is not merely a delivery format.

SPARK

Review links without hosting user assets

Export a self-contained “shot passport”: a local HTML viewer containing a compressed GLB, poster/video, camera cut list, credits and download button. The existing character passport proves the pattern.

SPARK

Peer review in the same room

WebRTC data channels can sync camera, laser pointer, annotations and approval notes peer-to-peer. Manual offer/answer QR/text pairing avoids accounts and servers. Start review-only; CRDT editing is much later.

PRO

OPFS project cache

Use Origin Private File System for big GLBs, thumbnails, footage and render frames, while IndexedDB retains the searchable project manifest. This is the natural storage foundation for film projects.

SPARK

WebXR rehearsal, not just viewing

AR already places a character. Next: VR scale check, controller-driven IK blocking, world-space camera scout, and recording controller trajectories as a camera take. It is a wonderful previsualization affordance.

PRO

File System Access workflows

Where supported, let a user open a project directory, autosave takes/renders beside source assets, and export a portable folder. Keep download/import as universal fallback.

PRO

Local models, but only as assistants

WASM/WebGPU inference can propose segmentation, depth, pose labels or motion cleanup. Model downloads must be explicit, offline-cacheable, size-labelled and optional; deterministic tools remain the core.

ARCHITECTURE BETS

Preserve the modular iframe model—but give it a project brain.

NOW

Project manifest v2

Today the lingua franca is GLB ArrayBuffer and the library is indexed locally. Add a versioned manifest that references assets, takes, expressions, poses, recipes, stage layouts, director sequences, material/texture-set metadata and export presets.

DO → Keep the manifest small JSON and migrations explicit. Save it with the existing .polypotion bundle. This is the least invasive way to make multi-tool work durable.
PRO

Shared worker as asset/service core—not a shared renderer

The existing audit correctly warns against rushing a shared mesh core. The incremental move is a SharedWorker/worker service responsible for asset byte cache, OPFS access, job scheduling, thumbnails and project manifest—not live Three.js scenes.

DO → Adopt stable IDs and immutable asset versions first. Tools can still receive transferred buffers, but now they ask the same core for the right version and save a new version instead of overwriting unknowingly.
FIX

Capability matrix and degradation policy

Browsers vary: threads/isolation, WebCodecs, WebGPU, XR, File System Access and codecs are not universal. A feature should declare its baseline, accelerated path, clear fallback and whether it changes output fidelity.

DO → Centralize feature probes in one platform module and show concise badges before an operation. Avoid scattered browser sniffing inside engines.
A SANE ORDER TO BREW

Do the compounding work before the spectacular work.

0 · Finish the floor small / medium

Vendor dependencies locally; move large blobs toward OPFS; add sample characters; finish older accessibility; enforce every heavy operation’s progress/cancel contract. This preserves the “free, offline, local” premise.

1 · Project manifest + compute jobs medium

Versioned project metadata, asset IDs, cancellable worker jobs and reusable task progress. This unlocks durable stages, takes and render queues without a dangerous rewrite.

2 · Performance takes + animation cleanup medium / large

Unify body, face and voice into editable takes; add smoothing, contacts, root motion, curve reduction and baking. This converts existing Mocap/Face/Lipsync/Pose tools into a coherent performer workflow.

3 · Small-film sequence + offline renderer large

Join Stage, Timeline, Director and audio under a compact sequence project; generalize Physics’ WebCodecs render path. The result is the clearest user-facing “wow” that is still tightly scoped.

4 · Corrective/constraint rig layer large

Pose-space correctives first, then FK/IK matching and spaces. It increases quality of everything downstream—mocap, posing, stage shots and exports.

5 · Pick one frontier experiment large / XL

Choose exactly one: VR camera/IK rehearsal; review-only local WebRTC; GPU SDF remesh; or scan-to-character Recipe. Do not build all four at once.

WHAT NOT TO DO

Guardrails that keep the project special

Keep local-first as a product feature, not a slogan.
Networked models/services can be opt-in, named and removable; users’ meshes, takes and work files must not be the price of admission.

Do not turn this into a generic 3D package

A full scene graph, arbitrary nodes, compositing, sculpting parity and a new renderer would dilute the advantage. Use imported props in Stage; invest in character performance and portable results.

Do not make WASM a religion

WASM is exceptional for hot loops and battle-tested native algorithms. It does not repair unclear UX, missing project state, weak cancellation, or data-model inconsistency. Benchmark, isolate, fallback.

Do not make AI the only path

AI suggestions should create editable regular data—selections, landmarks, keys, masks, takes—not opaque locked results. The deterministic manual tool must always remain viable.
